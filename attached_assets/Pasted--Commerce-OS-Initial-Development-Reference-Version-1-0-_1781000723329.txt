## 1. Project Overview

### Vision

A multi-tenant SaaS commerce platform enabling merchants to:

- Create and manage online stores
- Manage products, inventory, and orders
- Accept payments (local + global gateways)
- Run POS systems (offline-first, Flutter)
- Manage multiple branches and warehouses
- Run marketing campaigns and loyalty programs
- Manage customers (CRM)
- Track analytics and KPIs
- Connect logistics providers

**Equivalent to:** Shopify + POS + Inventory + CRM + Marketing + Logistics + Analytics

### Business Model

| Revenue Source | Details |
|---|---|
| Subscription plans | Starter, Growth, Business, Enterprise |
| Transaction fees | Percentage per order |
| SMS charges | Usage-based billing |
| Enterprise services | Custom development |
| Setup charges | Store onboarding |
| POS hardware | Scanner, printer, cash drawer |

## 3. Tenant & Multi-Tenancy Model

Every merchant is a **Tenant**. All data is scoped by `tenant_id`.

```
Tenant
├── Stores          (online storefronts, custom domains)
├── Warehouses      (stock locations)
├── Branches        (physical locations linked to warehouses)
├── Users           (RBAC, 7 roles)
├── Customers       (CRM, loyalty, segments)
├── Orders          (9-stage lifecycle)
├── Inventory       (6 stock states per variant per warehouse)
├── Themes          (store builder JSON, draft/published)
└── Domains         (custom per store, resolved via Host header)
```

### Isolation Strategy

- **Phase 1–5:** Shared tables with `tenant_id` column + PostgreSQL Row Level Security (RLS)
- **Enterprise tier:** Schema-per-tenant (`search_path` switching)
- **White-label / Phase 6:** Database-per-tenant (dedicated RDS instance)

### RLS Enforcement (Shared Tables)

```sql
-- Enable on every table
ALTER TABLE products ENABLE ROW LEVEL SECURITY;

-- Policy
CREATE POLICY tenant_isolation ON products
  USING (tenant_id = current_setting('app.tenant_id')::uuid);

-- Set at start of every transaction (via common-library middleware)
SET LOCAL app.tenant_id = '<uuid-from-jwt>';
```

**The `common-library` middleware must inject `tenant_id` from JWT on every request. No microservice queries without it.**

### Core Schema Highlights

#### tenants
```sql
CREATE TABLE tenants (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name          VARCHAR(255) NOT NULL,
  slug          VARCHAR(255) UNIQUE NOT NULL,
  plan          VARCHAR(50) NOT NULL DEFAULT 'starter',
  status        VARCHAR(50) NOT NULL DEFAULT 'active',
  created_at    TIMESTAMPTZ DEFAULT now()
);
```

#### products
```sql
CREATE TABLE products (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     UUID NOT NULL REFERENCES tenants(id),
  name          VARCHAR(255) NOT NULL,
  slug          VARCHAR(255) NOT NULL,
  description   TEXT,
  price         NUMERIC(12,2) NOT NULL,
  sku           VARCHAR(100),
  barcode       VARCHAR(100),
  status        VARCHAR(50) DEFAULT 'draft',
  visibility    VARCHAR(50) DEFAULT 'public',
  weight        NUMERIC(8,2),
  tax_class     VARCHAR(50),
  created_at    TIMESTAMPTZ DEFAULT now(),
  updated_at    TIMESTAMPTZ DEFAULT now(),
  UNIQUE(tenant_id, slug)
);
```

#### inventory
```sql
CREATE TABLE inventory (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     UUID NOT NULL REFERENCES tenants(id),
  variant_id    UUID NOT NULL REFERENCES variants(id),
  warehouse_id  UUID NOT NULL REFERENCES warehouses(id),
  available     INT NOT NULL DEFAULT 0,
  reserved      INT NOT NULL DEFAULT 0,  -- incremented on order.confirmed
  damaged       INT NOT NULL DEFAULT 0,
  returned      INT NOT NULL DEFAULT 0,
  lost          INT NOT NULL DEFAULT 0,
  on_hand       INT GENERATED ALWAYS AS (available + reserved + damaged + returned + lost) STORED,
  updated_at    TIMESTAMPTZ DEFAULT now(),
  UNIQUE(tenant_id, variant_id, warehouse_id)
);
```

#### inventory_transactions (ledger — never mutate inventory directly)
```sql
CREATE TABLE inventory_transactions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  inventory_id    UUID NOT NULL REFERENCES inventory(id),
  warehouse_id    UUID NOT NULL REFERENCES warehouses(id),
  type            VARCHAR(50) NOT NULL, -- purchase|sale|return|adjustment|transfer|damaged|lost
  quantity_delta  INT NOT NULL,
  reference_type  VARCHAR(50),          -- order|return|adjustment|transfer
  reference_id    UUID,
  created_by      UUID REFERENCES users(id),
  created_at      TIMESTAMPTZ DEFAULT now()
);
```

#### orders
```sql
CREATE TABLE orders (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  customer_id     UUID REFERENCES customers(id),
  status          VARCHAR(50) NOT NULL DEFAULT 'pending',
  -- pending|confirmed|packed|ready_to_ship|shipped|delivered|returned|refunded|cancelled
  subtotal        NUMERIC(12,2) NOT NULL,
  discount        NUMERIC(12,2) DEFAULT 0,
  shipping_fee    NUMERIC(12,2) DEFAULT 0,
  tax             NUMERIC(12,2) DEFAULT 0,
  total           NUMERIC(12,2) NOT NULL,
  currency        VARCHAR(10) DEFAULT 'NPR',
  notes           TEXT,
  created_at      TIMESTAMPTZ DEFAULT now(),
  updated_at      TIMESTAMPTZ DEFAULT now()
);
```

## 8. API Gateway Design

### Middleware Pipeline (executes in order on every request)

```
1. TLS termination + Cloudflare proxy
   └── HTTPS enforced, mTLS for internal service-to-service

2. Request ID injection
   └── X-Request-ID: UUIDv7 — generated here, propagated to all upstream services

3. Auth middleware
   └── Verify JWT signature
   └── Extract tenant_id, user_id, role, plan from claims
   └── Inject: X-Tenant-ID, X-User-ID, X-User-Role, X-Tenant-Plan headers
   └── Return 401 if invalid or expired

4. Tenant resolution
   └── If storefront: resolve tenant from Host header → Redis lookup → inject X-Tenant-ID
   └── If dashboard/API: tenant_id from JWT claim

5. Rate limiting (Redis sliding window)
   └── Starter:    100 req/min
   └── Growth:     500 req/min
   └── Business:  2000 req/min
   └── Enterprise: unlimited
   └── Return 429 on breach with Retry-After header

6. Route + proxy
   └── Strip /api/v1 prefix
   └── Forward to upstream service with injected headers
```

### Bypass Rules

```
/webhooks/*          → skip JWT, verify gateway HMAC signature instead
/storefront/public/* → skip JWT, rate-limit by IP only, resolve tenant from Host
/auth/login          → skip JWT (unauthenticated endpoint)
/auth/refresh        → skip JWT (uses refresh token cookie)
/health, /metrics    → IP whitelist to monitoring subnet only
```

### JWT Payload Structure

```json
{
  "sub": "user-uuid",
  "tenant_id": "tenant-uuid",
  "role": "MANAGER",
  "plan": "growth",
  "iat": 1700000000,
  "exp": 1700003600
}
```

## 9. Event-Driven Architecture (Kafka)

### Topic Naming Convention

`{domain}.{event}` — e.g. `order.confirmed`, `payment.success`

### Topic → Consumer Matrix

| Topic | Producers | Consumers |
|---|---|---|
| `order.created` | order-service | notification-service, analytics-service |
| `order.confirmed` | order-service | inventory-service (reserve stock), notification-service, analytics-service |
| `order.shipped` | order-service, shipment-service | notification-service, analytics-service |
| `order.delivered` | shipment-service | notification-service, analytics-service, marketing-service (loyalty) |
| `payment.success` | payment-service | order-service (confirm order), notification-service, analytics-service |
| `payment.failed` | payment-service | order-service (release reservation), notification-service |
| `product.created` | product-service | search-index (Elasticsearch), analytics-service |
| `product.updated` | product-service | search-index, analytics-service, theme-service (cache bust) |
| `inventory.changed` | inventory-service | notification-service (low stock alert), marketing-service, analytics-service |
| `customer.created` | customer-service | marketing-service, analytics-service |
| `notification.send` | any service | notification-service |

### Implementation Rules

- Every consumer must have a **dead letter queue** topic (e.g. `order.confirmed.dlq`)
- **Exactly-once semantics** for: `payment.success`, `payment.failed` (financial events)
- **At-least-once + idempotent consumers** for: analytics, notifications, search indexing
- Use **schema registry** (Confluent) from day one — version all event schemas
- Consumer group naming: `{service-name}-{topic}` e.g. `inventory-service-order.confirmed`

### Dead Letter Queue Pattern

```
order.confirmed
  → inventory-service (consumer group: inventory-service-order.confirmed)
      on failure after 3 retries → order.confirmed.dlq
          → dlq-processor (alert + manual review)
```

## 10. Order Lifecycle

```
PENDING          → payment initiated, stock NOT yet reserved
  ↓ (payment.success)
CONFIRMED        → payment verified, stock RESERVED (not decremented)
  ↓ (merchant action)
PACKED           → items picked, stock DECREMENTED, label generated
  ↓ (courier assigned)
READY TO SHIP    → courier assigned, awaiting pickup
  ↓ (courier pickup)
SHIPPED          → in transit, live tracking active
  ↓ (delivery confirmed)
DELIVERED        → POD captured, loyalty points added, stock reservation cleared
```

### Branch Flows

```
PENDING/CONFIRMED → CANCELLED    (auto-refund triggered if paid)
DELIVERED → RETURNED             (within return window, configurable per tenant)
RETURNED → REFUNDED              (gateway refund API called)
```

### Inventory Reservation Rules

- `order.confirmed` → `inventory.reserved += quantity` (never touch `available`)
- `order.packed` → `inventory.available -= quantity`, `inventory.reserved -= quantity`
- `order.cancelled` → `inventory.reserved -= quantity` (release reservation)
- `order.returned` → `inventory.returned += quantity` (pending inspection)

### Return Window

Configurable per tenant in `system_settings`. Default: 7 days after delivery. Enforced at API layer.

## 11. Inventory & Warehouse Model

### Stock States

| State | Meaning |
|---|---|
| `available` | Sellable stock — shown to customers |
| `reserved` | Held for confirmed orders not yet packed |
| `damaged` | Unusable, pending write-off |
| `returned` | Returned by customer, pending inspection |
| `lost` | Written off (theft, shrinkage) |
| `on_hand` | Computed: sum of all above |

### Inventory Movement Types

`purchase` · `sale` · `return` · `adjustment` · `transfer` · `damaged` · `lost`

### Warehouse Transfer

Two transaction rows per transfer:
- Source warehouse: `quantity_delta = -N`, `type = transfer`
- Destination warehouse: `quantity_delta = +N`, `type = transfer`
- Both rows share the same `reference_id` (transfer UUID)

### Branch → Warehouse Relationship

One branch draws from one warehouse. One warehouse can serve multiple branches. Filter `inventory` by `warehouse_id` for per-branch stock visibility.

## 12. Payment Gateway Integration

### Supported Gateways

| Gateway | Market | Flow Type |
|---|---|---|
| eSewa | Nepal | Redirect (user leaves site, returns with token) |
| Khalti | Nepal | Redirect |
| FonePay | Nepal | QR generation (merchant-side) |
| ConnectIPS | Nepal | Redirect |
| Stripe | Global | Intent API + webhook |
| PayPal | Global | Order API + webhook |

### Payment Flow

```
1. Customer submits checkout
2. Payment Service creates payment record (status: pending)
3. Route to gateway → redirect URL or QR returned to frontend
4. Customer completes payment on gateway
5. Gateway calls webhook → Payment Service /webhooks/{gateway}
6. Webhook handler verifies signature (HMAC) → parses event
7. On success:
   - Update payment record (status: paid)
   - Publish payment.success to Kafka
   - order-service consumes → confirms order
8. On failure:
   - Update payment record (status: failed)
   - Publish payment.failed to Kafka
   - order-service consumes → releases stock reservation
```

### Webhook Security

Each gateway uses a different verification method:

| Gateway | Method |
|---|---|
| eSewa | HMAC-SHA256 of payload with secret key |
| Khalti | Token-based verification via Khalti API |
| Stripe | `Stripe-Signature` header HMAC |
| PayPal | OAuth token verification |

**Never forward a webhook to the Payment Service without gateway-level signature verification at the API Gateway.**

### Idempotency

Store gateway `transaction_id` in `payments` table with a UNIQUE constraint. On duplicate webhook delivery: return `200 OK`, skip processing silently.

### Refund Abstraction

```typescript
interface RefundProvider {
  initiateRefund(transactionId: string, amount: number, reason: string): Promise<RefundResult>;
  getRefundStatus(refundId: string): Promise<RefundStatus>;
}
```

Each gateway implements this interface. The Order Service calls `PaymentService.refund()` — never a gateway directly.

## 13. POS Offline Sync Strategy

### Local Storage (Flutter SQLite)

```sql
-- Local device schema
CREATE TABLE local_products (
  id TEXT PRIMARY KEY,  -- UUID from server
  tenant_id TEXT,
  name TEXT,
  price REAL,
  barcode TEXT,
  updated_at TEXT
);

CREATE TABLE local_inventory (
  variant_id TEXT,
  warehouse_id TEXT,
  available INTEGER,
  last_synced_at TEXT,
  PRIMARY KEY (variant_id, warehouse_id)
);

CREATE TABLE pos_sales (
  id TEXT PRIMARY KEY,          -- UUIDv7 — client generated
  tenant_id TEXT,
  items_json TEXT,              -- JSON array of line items
  total REAL,
  payment_method TEXT,
  status TEXT DEFAULT 'pending_sync',
  created_at TEXT,
  synced_at TEXT
);

CREATE TABLE outbox (
  id TEXT PRIMARY KEY,          -- UUIDv7
  entity_type TEXT,             -- pos_sale|return|cash_in|cash_out
  payload_json TEXT,
  created_at TEXT,
  attempts INTEGER DEFAULT 0,
  last_attempt_at TEXT,
  error TEXT
);
```

### Sync Modes

**Online:** Initial full sync on app start, then WebSocket/SSE delta pushes from server.

**Offline:** All mutations write to `outbox` first, then to local SQLite. UUIDv7 for all client-generated IDs (time-ordered, no collision).

### Reconnect Sync Process

```
1. Drain outbox in created_at order (oldest first)
2. POST each entry to /api/v1/pos/sync
3. Server processes with idempotency (keyed on client UUID)
4. On success: mark outbox entry synced
5. On failure: exponential backoff (1s → 2s → 4s → … max 5min), max 10 retries → DLQ alert
6. After outbox drained: pull fresh delta snapshot (updated_at > last_synced_at)
7. Clear fully-synced outbox entries
```

### Conflict Resolution

| Conflict Type | Resolution |
|---|---|
| Inventory (sold offline, also sold online) | **Server wins.** Local sale accepted, cashier alerted, handled operationally |
| Sale (duplicate UUID) | **Idempotent upsert** — second delivery ignored |
| Price (offline price ≠ current server price > 5%) | **Flag for manager review** — do not auto-accept |
| Product update | **Server wins** — pull fresh on reconnect |

### Critical Rules

- **Never write to local state without an outbox entry first** — or mutations are silently lost
- **Delta sync only after initial snapshot** — pull `updated_at > last_synced_at`, never full sync on reconnect
- **Use UUIDv7** for all client-generated IDs (not UUIDv4 — ordering is critical)
- **Exponential backoff with jitter** on sync retries — cap at 5 min

### Draft vs Published

- Two JSON blobs per tenant: `draft` and `published`
- Merchants edit `draft` freely — no live impact
- Publish: atomically swaps `draft` → `published`, snapshots old `published` for rollback
- Redis cache keyed by `tenant_id + domain`, TTL 60s, invalidated on publish

### Storefront Rendering (Next.js SSR)

```typescript
// Per request:
// 1. Resolve tenant from Host header (Redis cache)
// 2. Fetch published theme JSON
// 3. Map section.type → React component
// 4. Render section tree with props

const componentMap: Record<string, React.FC> = {
  hero: HeroSection,
  product_grid: ProductGridSection,
  banner: BannerSection,
  // ...
};

export async function getServerSideProps({ req }) {
  const tenant = await resolveTenant(req.headers.host);
  const theme = await getPublishedTheme(tenant.id);
  return { props: { theme } };
}
```

## 15. Multi-Tenancy DB Isolation

### Strategy by Tier

| Tier | Strategy | When |
|---|---|---|
| Starter / Growth / Business | Shared tables + RLS | Phase 1–5, up to 10,000+ tenants |
| Enterprise | Schema per tenant (`search_path`) | ~100 tenant limit, strong isolation |
| White-label | Database per tenant (dedicated RDS) | Phase 6, maximum isolation |

### RLS Setup (Recommended — Phase 1–5)

```sql
-- 1. Enable on every table
ALTER TABLE {table} ENABLE ROW LEVEL SECURITY;
ALTER TABLE {table} FORCE ROW LEVEL SECURITY;

-- 2. Policy (using session variable set by app)
CREATE POLICY tenant_isolation ON {table}
  USING (tenant_id = current_setting('app.tenant_id')::uuid);

-- 3. App middleware (common-library) — runs on every request
-- Extracts tenant_id from JWT, sets session variable
await db.query("SET LOCAL app.tenant_id = $1", [tenantId]);
```

### common-library Middleware (mandatory)

```typescript
// Every microservice must use this middleware
// It must run before any database query

export async function tenantMiddleware(req, res, next) {
  const tenantId = req.headers['x-tenant-id']; // injected by API Gateway
  if (!tenantId) return res.status(400).json({ error: 'Missing tenant context' });

  await db.query('SET LOCAL app.tenant_id = $1', [tenantId]);
  req.tenantId = tenantId;
  next();
}
```

## 18. Security Requirements

| Requirement | Implementation |
|---|---|
| Authentication | JWT (access token 1h) + refresh token (30d, HTTP-only cookie) |
| Token rotation | Refresh token rotated on every use |
| MFA | TOTP (Google Authenticator) for TENANT_OWNER and MANAGER |
| Rate limiting | Per tenant per route via Redis sliding window in API Gateway |
| Audit logs | Every write operation logged to `audit_logs` with user, timestamp, before/after |
| IP whitelisting | `/health`, `/metrics` endpoints restricted to monitoring subnet |
| Encryption at rest | AWS RDS encryption, S3/MinIO server-side encryption |
| Encryption in transit | TLS 1.3 externally, mTLS internally (Istio service mesh) |
| Secrets management | AWS Secrets Manager, never in env files or source code |
| Dependency scanning | Trivy in CI on every build |
| RBAC | Role embedded in JWT, enforced at API Gateway + service layer |
| Webhook security | HMAC signature verification at API Gateway before forwarding |
| SQL injection | Parameterised queries only — no string concatenation in SQL |
| XSS | Content-Security-Policy headers, output encoding in storefront |

## 25. Subscription Billing Model

### Plans & Limits

| Feature | Starter | Growth | Business | Enterprise |
|---|---|---|---|---|
| Monthly price (NPR) | 999 | 2,999 | 7,999 | Custom |
| Products | 100 | 1,000 | Unlimited | Unlimited |
| Orders/month | 500 | 5,000 | Unlimited | Unlimited |
| Staff accounts | 2 | 5 | 20 | Unlimited |
| Branches | 1 | 3 | 10 | Unlimited |
| Custom domain | ✗ | ✓ | ✓ | ✓ |
| POS terminals | 1 | 3 | 10 | Unlimited |
| SMS included/month | 100 | 500 | 2,000 | Custom |
| API rate limit (req/min) | 100 | 500 | 2,000 | Unlimited |
| Analytics retention | 30d | 90d | 1yr | Custom |
| White-label | ✗ | ✗ | ✗ | ✓ |

### Transaction Fees

| Plan | Transaction Fee |
|---|---|
| Starter | 2.0% per order |
| Growth | 1.5% per order |
| Business | 1.0% per order |
| Enterprise | 0% (flat fee) |

### Billing Cycle

```
1. Monthly subscription charged on billing_date (day of month tenant signed up)
2. Transaction fees accrued daily, invoiced monthly
3. SMS charges billed at end of month (usage × per-SMS rate)
4. Overage charges (e.g. orders > plan limit) billed at next cycle
```

### Database Schema

```sql
CREATE TABLE plans (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name        VARCHAR(50) UNIQUE NOT NULL,  -- starter|growth|business|enterprise
  price_npr   NUMERIC(10,2) NOT NULL,
  limits      JSONB NOT NULL,               -- {products, orders, staff, branches, ...}
  txn_fee_pct NUMERIC(5,4) NOT NULL,
  created_at  TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE subscriptions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  plan_id         UUID NOT NULL REFERENCES plans(id),
  status          VARCHAR(50) DEFAULT 'active', -- active|past_due|cancelled|trialing
  billing_date    INT NOT NULL,                 -- day of month (1-28)
  trial_ends_at   TIMESTAMPTZ,
  current_period_start TIMESTAMPTZ NOT NULL,
  current_period_end   TIMESTAMPTZ NOT NULL,
  cancelled_at    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE invoices (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  subscription_id UUID NOT NULL REFERENCES subscriptions(id),
  period_start    TIMESTAMPTZ NOT NULL,
  period_end      TIMESTAMPTZ NOT NULL,
  subtotal        NUMERIC(12,2) NOT NULL,
  tax             NUMERIC(12,2) DEFAULT 0,
  total           NUMERIC(12,2) NOT NULL,
  status          VARCHAR(50) DEFAULT 'pending', -- pending|paid|overdue|void
  paid_at         TIMESTAMPTZ,
  due_date        TIMESTAMPTZ NOT NULL,
  line_items      JSONB NOT NULL,               -- [{description, quantity, unit_price, total}]
  created_at      TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE usage_records (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL REFERENCES tenants(id),
  type        VARCHAR(50) NOT NULL,  -- sms|transaction_fee|api_call|storage_gb
  quantity    NUMERIC(12,4) NOT NULL,
  unit_price  NUMERIC(10,6) NOT NULL,
  reference_id UUID,
  recorded_at TIMESTAMPTZ DEFAULT now()
);
```

### Billing Flow

```
1. Cron job runs daily at midnight
2. Check subscriptions where current_period_end <= now()
3. Calculate invoice:
   a. Base plan price
   b. Sum usage_records for period (SMS, transaction fees, overages)
   c. Apply any discounts/coupons
4. Generate invoice record
5. Charge via stored payment method
6. On success: advance current_period_start/end, publish subscription.renewed event
7. On failure: mark past_due, send dunning email, retry after 3/7/14 days
8. After 3 failed retries: suspend tenant (read-only mode), send final notice
```

### Plan Enforcement

Plan limits are enforced at the API layer via the `common-library` plan guard:

```typescript
// Applied as a NestJS guard on write endpoints
async canActivate(context): Promise<boolean> {
  const tenant = await this.tenantService.findById(req.tenantId);
  const limits = tenant.plan.limits;

  if (endpoint === 'POST /products') {
    const count = await this.productRepo.countByTenant(req.tenantId);
    if (count >= limits.products) throw new ForbiddenException('Plan limit reached');
  }
  return true;
}
```

### Cache-Aside Pattern (standard)

```typescript
async getProduct(tenantId: string, productId: string): Promise<Product> {
  const cacheKey = `${tenantId}:product:${productId}`;

  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);

  const product = await db.query(
    'SELECT * FROM products WHERE tenant_id = $1 AND id = $2',
    [tenantId, productId]
  );

  if (product) await redis.setex(cacheKey, 300, JSON.stringify(product));
  return product;
}
```

### Cache Invalidation via Kafka

```typescript
// In notification/cache-invalidator service

kafkaConsumer.on('product.updated', async (event) => {
  await redis.del(`${event.tenant_id}:product:${event.product_id}`);
```

### Session Storage

JWT access tokens are **stateless** — no Redis needed for validation.
Refresh tokens are stored in Redis with the user ID as value:

```
refresh:{token_hash}  →  { user_id, tenant_id, issued_at }  TTL: 30d
```

On logout: `DEL refresh:{token_hash}` — immediate revocation.

### Receipt Printing

Receipts are generated as ESC/POS commands sent to the connected thermal printer via Bluetooth or USB. The Flutter POS app handles this natively.

Required on every receipt:
- Store name, address, phone
- Register name + cashier name
- Date + time
- Item list (name, qty, unit price, line total)
- Subtotal, discount, tax, total
- Payment method + amount tendered + change
- Order/sale ID (for reference)
- Loyalty points earned (if applicable)
- Footer: thank you message + return policy

### Cash Management

```
Open Register:    Record opening_cash, set status = 'open'
Cash In:          Log to cash_movements (type=in), update expected_cash
Cash Out:         Log to cash_movements (type=out), update expected_cash
Close Register:   Enter closing_cash, compute variance, set status = 'closed'

Expected cash = opening_cash + sum(cash_in) - sum(cash_out) + sum(cash_sales) - sum(cash_refunds)
```

### BaseRepository Pattern

```typescript
export abstract class BaseRepository<T> {
  constructor(protected db: Pool, protected tableName: string) {}

  async findById(tenantId: string, id: string): Promise<T | null> {
    const { rows } = await this.db.query(
      `SELECT * FROM ${this.tableName} WHERE id = $1`,
      [id]  // RLS handles tenant_id filtering via session var
    );
    return rows[0] ?? null;
  }

  async findAll(tenantId: string, opts: PaginationOpts): Promise<PaginatedResult<T>> {
    const { rows } = await this.db.query(
      `SELECT * FROM ${this.tableName} ORDER BY created_at DESC LIMIT $1 OFFSET $2`,
      [opts.limit, opts.offset]
    );
    const { rows: [{ count }] } = await this.db.query(
      `SELECT COUNT(*) FROM ${this.tableName}`
    );
    return { data: rows, total: parseInt(count), ...opts };
  }

  async create(tenantId: string, data: Partial<T>): Promise<T> {
    const id = generateUUIDv7();
    const fields = Object.keys(data);
    const values = Object.values(data);
    const placeholders = fields.map((_, i) => `$${i + 2}`);

    const { rows } = await this.db.query(
      `INSERT INTO ${this.tableName} (id, tenant_id, ${fields.join(', ')})
       VALUES ($1, ${tenantId}, ${placeholders.join(', ')}) RETURNING *`,
      [id, ...values]
    );
    return rows[0];
  }
}
```

```typescript
// src/orders/orders.service.ts
import { Injectable, NotFoundException, BadRequestException } from '@nestjs/common';
import { OrdersRepository } from './orders.repository';
import { publishEvent, logger, withSpan } from '@commerce-os/common-library';
import { OrderStatus } from './entities/order.entity';

const VALID_TRANSITIONS: Record<OrderStatus, OrderStatus[]> = {
  pending:        ['confirmed', 'cancelled'],
  confirmed:      ['packed', 'cancelled'],
  packed:         ['ready_to_ship'],
  ready_to_ship:  ['shipped'],
  shipped:        ['delivered'],
  delivered:      ['returned'],
  returned:       ['refunded'],
  refunded:       [],
  cancelled:      [],
};

@Injectable()
export class OrdersService {
  constructor(private readonly repo: OrdersRepository) {}

  async create(tenantId: string, userId: string, dto: CreateOrderDto) {
    return withSpan('orders.create', async (span) => {
      const order = await this.repo.create(tenantId, { ...dto, status: 'pending' });
      span.setAttribute('order.id', order.id);

      await publishEvent('order.created', {
        tenant_id: tenantId,
        order_id: order.id,
        customer_id: order.customer_id,
        total: order.total,
      });

      logger.info('Order created', { tenant_id: tenantId, order_id: order.id });
      return order;
    });
  }

  async updateStatus(tenantId: string, orderId: string, newStatus: OrderStatus, userId: string) {
    const order = await this.repo.findById(tenantId, orderId);
    if (!order) throw new NotFoundException('Order not found');

    const allowed = VALID_TRANSITIONS[order.status];
    if (!allowed.includes(newStatus)) {
      throw new BadRequestException(
        `Cannot transition from ${order.status} to ${newStatus}`
      );
    }

    const updated = await this.repo.updateStatus(tenantId, orderId, newStatus);

    await publishEvent(`order.${newStatus}`, {
      tenant_id: tenantId,
      order_id: orderId,
      previous_status: order.status,
      new_status: newStatus,
      updated_by: userId,
    });

    return updated;
  }
}
```

## 37. Architecture Decision Records (ADRs)

### ADR-001: UUIDv7 for All Primary Keys

**Status:** Accepted

**Context:** Need primary keys that are globally unique, sortable by creation time (for efficient index inserts), and generatable on the client (POS offline).

**Decision:** Use UUIDv7 (RFC 9562) for all primary keys across all services and client-generated IDs.

**Consequences:** Time-ordered inserts are efficient on B-tree indexes. Client can generate IDs without a round-trip. No auto-increment sequences to manage across services.

---

### ADR-002: PostgreSQL RLS for Tenant Isolation (Phase 1–5)

**Status:** Accepted

**Context:** Need tenant data isolation. Options: application-layer filtering, RLS, schema-per-tenant, DB-per-tenant.

**Decision:** PostgreSQL Row Level Security with `SET LOCAL app.tenant_id` injected by `common-library` middleware on every transaction.

**Consequences:** Strong isolation enforced at DB layer — even a bug in application code cannot leak cross-tenant data. Requires discipline to always use `common-library` middleware. Adds ~1ms per query for RLS policy evaluation.

---

### ADR-003: Outbox Pattern for POS Offline Mutations

**Status:** Accepted

**Context:** POS Flutter app must operate offline and sync mutations reliably when connectivity returns. Risk: mutations written to local state but lost before sync.

**Decision:** All local mutations write to an `outbox` table first (within the same SQLite transaction). Sync engine drains outbox in creation order.

**Consequences:** Mutations are never lost even on app crash. Outbox becomes the source of truth for sync state. Adds complexity to all local write operations.

---

### ADR-004: Theme as JSON Schema (Not Code)

**Status:** Accepted

**Context:** Store builder needs to allow merchants to customise their storefront without writing code. Options: render arbitrary HTML/CSS, page builder generating code, JSON-driven section renderer.

**Decision:** Theme is a structured JSON document. The Next.js storefront maps section types to pre-built React components. Merchants configure props, not code.

**Consequences:** Safe — no arbitrary code execution. Predictable — merchants can't break their store. Limited — only supported section types are available (by design). Adding new section types is a platform release, not a merchant action.

---

### ADR-005: Kafka for All Inter-Service Events

**Status:** Accepted

**Context:** Services need to react to events from other services (e.g. payment confirmed → confirm order). Options: direct HTTP calls, message queue (RabbitMQ), event streaming (Kafka).

**Decision:** Kafka for all inter-service events. Services communicate exclusively via events for async operations. Synchronous HTTP only for queries that need an immediate response.

**Consequences:** Services are fully decoupled. Replay is possible (Kafka log retention). At-scale throughput is excellent. Adds operational complexity (Kafka cluster). Dead letter queues are mandatory.

---

### ADR-006: common-library as Internal NPM Package

**Status:** Accepted

**Context:** 16 services share auth middleware, logging, RLS setup, Kafka client, etc. Options: copy-paste per service, shared library, API-based shared service.

**Decision:** `common-library` published as a private NPM package (`@commerce-os/common-library`), versioned with semver. All services declare it as a dependency.

**Consequences:** Single place to update observability, auth, RLS. Breaking changes require major version bump and coordinated upgrade. Must be well-tested — a bug here affects all services.

## 38. Runbooks

### RB-001: Order Stuck in Pending

**Symptoms:** Customer complains order not confirmed after payment. Order status = `pending`.

**Investigation:**
```bash
# Check payment record
SELECT * FROM payments WHERE order_id = '<order_id>';

# Check Kafka consumer lag for payment.success topic
kafka-consumer-groups --bootstrap-server kafka:9092 --describe --group order-service-payment.success

# Check order-service logs
kubectl logs -n commerce-core -l app=order-service --since=1h | grep '<order_id>'
```

**Resolution:**
1. If payment record shows `status = paid` but order is `pending` → consumer lag or processing failure. Check DLQ: `payment.success.dlq`
2. If payment record shows `status = pending` → webhook not received. Manually trigger gateway reconciliation endpoint.
3. If payment record missing → payment was never initiated. Contact merchant.

---

### RB-002: High Kafka Consumer Lag

**Symptoms:** Grafana alert fires: `kafka_consumer_group_lag > 10000`

**Investigation:**
```bash
# Identify which consumer group and topic
kubectl exec -n monitoring kafka-0 -- \
  kafka-consumer-groups --bootstrap-server localhost:9092 --describe --all-groups

# Check consumer pod health
kubectl get pods -n commerce-core -l app=<service-name>
kubectl logs -n commerce-core <pod-name> --tail=100
```

**Resolution:**
1. If pods are healthy: likely a processing spike. Monitor — lag should self-resolve.
2. If pods are crash-looping: check logs for errors. Fix bug, redeploy.
3. If lag is growing without bound: scale up consumer replicas.
   ```bash
   kubectl scale deployment <service-name> -n commerce-core --replicas=5
   ```

### RB-003: Tenant Locked Out (Past Due)

**Symptoms:** Merchant reports they can't access dashboard. API returns 402.

**Investigation:**
```sql
SELECT s.status, s.current_period_end, i.status as invoice_status, i.due_date
FROM subscriptions s
JOIN invoices i ON i.subscription_id = s.id
WHERE s.tenant_id = '<tenant_id>'
ORDER BY i.created_at DESC LIMIT 5;
```

**Resolution:**
1. If legitimate past_due: merchant must pay via billing portal.
2. If payment processing error (e.g. gateway timeout): manually mark invoice as paid after confirming payment, update subscription status to `active`.
3. Emergency override (e.g. grace period): `UPDATE subscriptions SET status = 'active', current_period_end = now() + interval '7 days' WHERE tenant_id = '<id>';`

### RB-004: POS Sync Failing

**Symptoms:** Cashier reports sales not syncing. POS shows pending mutations.

**Investigation:**
```bash
# Check pos-service health
kubectl get pods -n commerce-core -l app=pos-service

# Check pos-service logs for the tenant
kubectl logs -n commerce-core -l app=pos-service --since=1h | grep '<tenant_id>'

# Check for duplicate UUID conflicts
```

**Resolution:**

## 42. Discount Rules Engine

The discount rules engine handles complex pricing logic beyond simple coupon codes. It evaluates a stack of discount rules in priority order and applies the best valid combination.

### Rule Types

| Type | Description | Example |
|---|---|---|
| `percentage_off` | % off cart or specific items | 20% off all shoes |
| `fixed_amount_off` | Fixed NPR off cart | Rs 200 off orders > Rs 2000 |
| `free_shipping` | Waive shipping fee | Free shipping over Rs 999 |
| `buy_x_get_y` | Buy N items, get M free/discounted | Buy 2 get 1 free |
| `bundle` | Discount when specific products bought together | Phone + case = 15% off |
| `tiered` | Discount scales with quantity | Buy 3 = 5%, buy 5 = 10%, buy 10 = 20% |
| `flash_sale` | Time-limited price override | 50% off for next 2 hours |
| `loyalty_discount` | Auto-applied for loyalty tier | Gold members get 10% off always |

### Rule Schema

```sql
CREATE TABLE discount_rules (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  name            VARCHAR(100) NOT NULL,
  type            VARCHAR(50) NOT NULL,
  priority        INT NOT NULL DEFAULT 0,    -- lower = applied first
  is_active       BOOLEAN DEFAULT true,
  is_stackable    BOOLEAN DEFAULT false,      -- can combine with other rules?
  conditions      JSONB NOT NULL,             -- when this rule triggers
  actions         JSONB NOT NULL,             -- what discount to apply
  starts_at       TIMESTAMPTZ,
  expires_at      TIMESTAMPTZ,
  usage_limit     INT,
  usage_count     INT DEFAULT 0,
  created_at      TIMESTAMPTZ DEFAULT now()
);
```

**Conditions JSON structure:**

```json
{
  "operator": "AND",
  "rules": [
    { "field": "cart.subtotal",    "op": "gte", "value": 1000 },
    { "field": "cart.item_count",  "op": "gte", "value": 2 },
    { "field": "customer.tier",    "op": "in",  "value": ["gold", "platinum"] },
    { "field": "product.category", "op": "in",  "value": ["uuid-shoes", "uuid-bags"] },
    { "field": "time.current",     "op": "between", "value": ["09:00", "18:00"] }
  ]
}
```

**Actions JSON structure:**

```json
{
  "type": "percentage_off",
  "value": 20,
  "applies_to": "matching_items",   // cart | matching_items | cheapest_item | shipping
  "max_discount_npr": 500,          // cap
  "free_items": 1                   // for buy_x_get_y
}
```

### Rule Evaluation Engine

```typescript
class DiscountEngine {
  async evaluate(cart: Cart, customer: Customer | null, tenantId: string): Promise<DiscountResult> {
    // 1. Load all active rules for tenant, sorted by priority
    const rules = await this.loadActiveRules(tenantId);

    // 2. Evaluate each rule's conditions against cart + customer context
    const eligible = rules.filter(rule => this.matchesConditions(rule.conditions, { cart, customer }));

    // 3. Separate stackable vs non-stackable
    const nonStackable = eligible.filter(r => !r.is_stackable);
    const stackable    = eligible.filter(r => r.is_stackable);

    // 4. Apply best non-stackable rule (highest discount value)
    const bestRule = this.findBestRule(nonStackable, cart);

    // 5. Apply all stackable rules on top
    const applied = [bestRule, ...stackable].filter(Boolean);

    // 6. Calculate final discounts
    return this.applyRules(applied, cart);
  }

  private matchesConditions(conditions: ConditionGroup, context: EvalContext): boolean {
    const { operator, rules } = conditions;
    const results = rules.map(rule => this.evaluateCondition(rule, context));
    return operator === 'AND' ? results.every(Boolean) : results.some(Boolean);
  }
}
```

```typescript
// Define job
interface GenerateInvoiceJob {

// Register worker
billingQueue.process('generate_invoice', async (job: Job<GenerateInvoiceJob>) => {

});
```

## 44. Analytics Data Warehouse

For real-time operational analytics, the `analytics_events` PostgreSQL table (partitioned by month) handles queries up to Phase 4. From Phase 5 onward, a dedicated data warehouse handles historical analysis, cohort queries, and cross-tenant reporting.

### Architecture

```
Kafka analytics_events topic
  ↓
Kafka Connect (JDBC sink / custom connector)
  ↓
AWS Redshift or ClickHouse (data warehouse)
  ↓
Grafana (BI dashboards) + dbt (transformations)
```

### ClickHouse (Recommended over Redshift for this scale)

ClickHouse is preferred because:
- Columnar storage — 10–100× faster for aggregation queries
- Handles billions of rows on modest hardware
- Excellent Kafka consumer integration
- Real-time inserts without batching delays
- Self-hosted or ClickHouse Cloud

### Warehouse Schema (ClickHouse)

```sql
-- Event fact table
CREATE TABLE analytics.events (

  variant_id    Nullable(UUID),

PARTITION BY (tenant_id, toYYYYMM(occurred_at))
```

## 45. Referral System

### How It Works

```
1. Existing customer gets unique referral code (auto-generated on account creation)
2. Customer shares code / referral link: https://store.com/?ref=ABC123
3. New customer clicks link → referral_code stored in cookie/session
4. New customer completes first order
5. On order.delivered:
   a. Referee (new customer) receives discount on next order
   b. Referrer (existing customer) earns loyalty points or store credit
```

### Schema

```sql
CREATE TABLE referral_codes (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id   UUID NOT NULL REFERENCES tenants(id),
  customer_id UUID NOT NULL REFERENCES customers(id),
  code        VARCHAR(20) UNIQUE NOT NULL,
  uses_count  INT DEFAULT 0,
  total_earned NUMERIC(12,2) DEFAULT 0,
  created_at  TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE referrals (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id       UUID NOT NULL REFERENCES tenants(id),
  referral_code_id UUID NOT NULL REFERENCES referral_codes(id),
  referrer_id     UUID NOT NULL REFERENCES customers(id),
  referee_id      UUID NOT NULL REFERENCES customers(id),
  order_id        UUID REFERENCES orders(id),
  status          VARCHAR(50) DEFAULT 'pending',  -- pending|qualified|rewarded|expired
  referrer_reward JSONB,   -- {type: 'points'|'credit', value: 200}
  referee_reward  JSONB,   -- {type: 'coupon', code: 'REF10', discount: 10}
  created_at      TIMESTAMPTZ DEFAULT now(),
  qualified_at    TIMESTAMPTZ
);
```
